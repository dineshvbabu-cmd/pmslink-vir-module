01
Module Scope & Design Principles
What VIR covers, key differentiators vs generic QHSE tools
Core Purpose
  • Full lifecycle management of all vessel inspections — from scheduling through to close-out
  • Questionnaire-based inspection with conditional logic, risk levels, and mandatory flagging
  • Observation / finding capture with photo evidence (offline capable)
  • PSC Concentrated Inspection Campaign (CIC) auto-detection and highlight
  • Defect carryover between consecutive VIRs until formally closed
  • AI-powered import of third-party PDF reports (Idwal, Anglo-Eastern, RightShip, etc.)
  • Dual sign-off: Vessel → Shore Review → Acknowledgement → Close
  • Gantt-based inspection calendar across fleet
Design Differentiators vs Generic QHSE
  • JiBe-parity UX — Section grouping, color-coded findings, photo timeline
  • CIC Intelligence — Auto-flag concentrated PSC items in questionnaire view
  • VIR-to-VIR continuity — Unresolved findings auto-surface in next inspection
  • Shore gate — Vessel cannot close VIR without shore review and feedback logged
  • Reference images — Small thumbnail standards; actual evidence uploads are full-size
  • AI import — Drop a PDF; OpenAI maps it to PMSLink VIR structure
  • True offline — Photos compressed and queued; full sync on reconnect
Note: VIR is a separate module from the existing QHSE checklist module. It replaces ad-hoc PDF/Word-based inspection recording currently done offline. The existing QHSE module retains Safety Meetings, Drills, and Non-Conformance tracking.

Offline & Sync Architecture
VESSEL (OFFLINE) IndexedDB stores: Draft VIR header, questionnaire responses, observations, photo queue (base64 compressed WebP)
↓ Auto-detect connectivity
SYNC TRIGGER Service Worker detects online → Background Sync API fires → Queued records POST to API
↓ Server receives
API LAYER Idempotent endpoints (ClientId dedup). Photos compressed server-side → Blob. Conflict resolution: last-write-wins for responses, merge for observations.
↓ Notification
SHORE OFFICE Real-time SignalR push on sync complete. Shore reviewer notified for dual sign-off gate.
Photo Handling Policy
  • Capture: Native camera via Capacitor (mobile) or file input (tablet/desktop)
  • Compression: WebP, max 1200px wide, quality 75% before IndexedDB store
  • Reference images: Max 300px thumbnails stored in DB as blob preview
  • Evidence uploads: Full-size stored in Blob; thumbnail generated server-side
  • Offline queue: Up to 200 photos queued; sync on reconnect in batches of 10
  • EXIF: GPS and device metadata stripped before upload for privacy

CIC Auto-Highlight Logic

// .NET API — CIC detection on VIR create public async Task<VirDto> CreateInspection(CreateVirCmd cmd) { var vir = new VirInspection { ... }; if (cmd.InspectionType == InspType.PortStateControl) { var cicYear = DateTime.Now.Year; var cicItems = await _cicRepo .GetActiveItems(cicYear); // Flag questionnaire items matching CIC topics foreach (var q in vir.Questions) { q.IsCICItem = cicItems .Any(c => c.TopicKeywords .Any(k => q.Text .Contains(k, StringComparison.OrdinalIgnoreCase))); } // Sort CIC items to top within each section vir.Questions = vir.Questions .OrderByDescending(q => q.IsCICItem) .ThenBy(q => q.SeqNo) .ToList(); } return Map(vir); }
Mandatory vs Non-Mandatory: Mandatory questions (IsMandatory = true) display a REQUIRED badge. Attempting to submit with unanswered mandatory HIGH-risk questions triggers a blocking modal listing outstanding items.
Inspection Standard Reference: Each question can display its source reference (e.g., SIRE 2.0 – 10.4.1, CDI Core-6.1.12, TMSA Element 4) as a collapsible info pane. Inspectors tap to see full clause text.

Defect Carryover Rules

VIR CLOSE GATE Before closing, system asks: "Were all observations from previous VIR [ID] closed?" Shows table of open items.
↓ If any remain open
CARRYOVER Open findings auto-copied to new VIR with IsCarriedOver=true and CarriedFromObsId link. Badge shows CARRIED OVER.
↓ Inspector updates
CLOSE CARRIED ITEM Inspector marks defect closed (requires evidence photo + close remark). Propagates closure back to original observation record.

Corrective Action Workflow

Two-Question Close Rule: Before a corrective action can be marked complete, the system prompts: (1) "Has the defect been physically rectified?" and (2) "Has the evidence been photographed and uploaded?" Both must be confirmed Yes.

Reference Image vs Evidence Upload

Reference images are set by office on the questionnaire template — small standard thumbnails (max 300px) showing what "good" looks like. These are view-only.

Evidence uploads by vessel are full-size (up to 8MB raw, compressed to ~600KB WebP). The viewer shows them at 800px with a lightbox for full-size review.

07
Dual Sign-Off Workflow
3-stage workflow: Vessel Submit → Shore Review → Final Close. Shore gate is mandatory.
STAGE 1 — VESSEL Inspector completes all sections. Mandatory items checked. Vessel Master or CE reviews. Signs off digitally. Status → SUBMITTED
→
STAGE 2 — SHORE REVIEW QHSE/Fleet Mgr notified. Reviews all findings and photos. Adds shore feedback comments per observation. Approves or returns with comments. Status → SHORE REVIEWED
→
STAGE 2b — RETURNED If shore returns: Vessel sees shore comments highlighted in red per item. Must address and resubmit. Status → RETURNED
→
STAGE 3 — CLOSE After shore approval, VIR can be closed. Vessel acknowledges shore feedback. System checks all CAs have completion date. Status → CLOSED
Constraint: System enforces that VIR cannot be set to CLOSED without Stage 2 shore sign-off logged. This is a backend validation — bypassing via API is not possible. Shore sign-off creates a VIR_SignOff record with Stage=2 and IsApproved=true.

AI-Powered PDF Import Engine
OpenAI GPT-4o converts third-party inspection PDFs into PMSLink VIR data structure

Import Flow

1 — UPLOAD User uploads PDF via web or mobile (max 50MB). File stored in blob. Import job queued via Hangfire.
↓
2 — EXTRACT .NET extracts text using PdfPig library. Images extracted via Aspose.PDF or iTextSharp. Text chunked into sections ≤ 4,000 tokens.
↓
3 — AI PARSE GPT-4o called with structured prompt. Returns JSON matching VIR schema: header fields, findings array, corrective actions array.
↓
4 — REVIEW SCREEN User reviews AI-parsed data side-by-side with original PDF. Edit, confirm, or reject individual fields. Confidence score shown per field.
↓
5 — COMMIT User confirms. VIR record created with ImportSourceId. Status = IMPORTED (sub-status before Draft).

GPT-4o Prompt Structure

// System prompt (trimmed) "You are a maritime inspection data extractor. Extract structured data from the following vessel inspection report text. Return ONLY valid JSON matching this schema: { header: { vesselName, inspectionDate, inspectorName, inspectorOrg, port, inspectionType, nonConformities, observations, recommendations }, findings: [{ findingType, priority, title, description, suggestedAction, dueDate, references, category }], correctiveActions: [{ observationIndex, actionRef, title, description, dueDate, status, remarks }] } Use inspectionType from this list: [SIRE, SIRE2, PSC, CDI, RightShip, InternalOwner, ...]. For findingType: NC | Observation | Recommendation | Positive. For priority: Critical | High | Medium | Low. Return null for fields not found. Do not invent data."
Token Strategy: For long reports (60+ pages like the sample Marine Hope INI-O-854), text is split into 5–8 chunks. Header extraction uses pages 1–2. Each findings section processed separately. Results merged client-side with dedup on finding title hash.
Supported Source Formats
  • Anglo-Eastern / Idwal Vessel Manager Reports
  • RightShip GHIA inspection PDFs
  • SIRE 2.0 questionnaire exports
  • ClassNK / BV / DNV survey reports
  • Generic inspection PDFs (best-effort)
  • Scanned PDFs → OCR via Azure Form Recognizer pre-process

.NET API — Import Controller Sketch

// POST /api/vir/import [HttpPost("import")] public async Task<IActionResult> ImportPdf([FromForm] IFormFile file, [FromForm] int vesselId) { var blobUrl = await _blobService.UploadAsync(file); var jobId = BackgroundJob.Enqueue<VirImportJob>( j => j.ProcessAsync(blobUrl, vesselId, UserId)); return Accepted(new { jobId }); } // VirImportJob.ProcessAsync public async Task ProcessAsync(string blobUrl, int vesselId, string userId) { var text = await _pdfExtractor.ExtractTextAsync(blobUrl); var chunks = TextChunker.Split(text, maxTokens: 3500); var results = new List<VirImportDto>(); foreach (var chunk in chunks) { var response = await _openAI.ChatAsync( systemPrompt: VirPrompts.ExtractionPrompt, userContent: chunk); var dto = JsonSerializer.Deserialize<VirImportDto>(response); results.Add(dto); } var merged = VirImportMerger.Merge(results); await _importRepo.SaveDraftAsync(merged, vesselId, blobUrl); await _hub.SendAsync(userId, "ImportComplete", merged.DraftId); }

Demo Build Guide (VB.NET / ASP.NET)
Build a standalone demo using Visual Basic .NET + ASP.NET Web Forms or MVC — no Angular needed for the demo
Demo Scope (What to Build)
  • Demo 1: PDF Import — Upload the Marine Hope INI-O-854 PDF → call OpenAI API → display parsed VIR header + findings in a grid. Show side-by-side PDF vs extracted data.
  • Demo 2: VIR Form — Simple ASP.NET form with vessel, inspection type, date, port fields. Add observation. Add corrective action. Show status.
  • Demo 3: CIC Highlight — Select "Port State Control", see questionnaire with 2–3 items highlighted in amber as CIC concentrated.
  • Demo 4: PDF Export — Button generates a styled PDF from demo data using iTextSharp in VB.NET.
Tech Choice: Use ASP.NET MVC 5 (not Core) with VB.NET code-behind for demo — simplest to run locally. Single project, SQL LocalDB, no Docker needed. OpenAI called from VB.NET using HttpClient with the REST API directly (no SDK needed).

VB.NET OpenAI Call Snippet

' VB.NET — Call OpenAI GPT-4o for PDF parsing Public Async Function ParsePdfText(pdfText As String) _ As Task(Of String) Dim client = New HttpClient() client.DefaultRequestHeaders.Add( "Authorization", "Bearer " & _apiKey) Dim body = JsonSerializer.Serialize(New With { .model = "gpt-4o", .max_tokens = 2000, .messages = New() { New With { .role = "system", .content = VirPrompts.ExtractionPrompt }, New With { .role = "user", .content = pdfText } } }) Dim response = Await client.PostAsync( "https://api.openai.com/v1/chat/completions", New StringContent(body, Encoding.UTF8, "application/json")) Dim json = Await response.Content.ReadAsStringAsync() Dim result = JsonDocument.Parse(json) Return result.RootElement .GetProperty("choices")(0) .GetProperty("message") .GetProperty("content") .GetString() End Function
